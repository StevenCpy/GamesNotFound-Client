Day 1
- Created Vite project.
- Created main navigation bar containing Home and Store pages.

What I learned today:
- From Vite, Parcel and rsbuild, do not use Parcel.  Vite is industry-standard but rsbuild is faster.
- To create Vite project, use "npm create vite@latest ." with JavaScript + React Compiler flag.  To run use "npm run dev".
- main.jsx vs App.jsx.  main.jsx is app entry point and App.jsx contains logic and structure.  main.jsx rarely modified.

Day 2
- Created StoreGame component to hold info about each game

What I learned today:
- Component function takes only 1 parameter which is an array, i.e. func({ param1, param2 }) instead of func(param1, param2)
- props are arguments in React and passed as follows: <StoreGame gameName="myFirstGame" gameVersion="1.0" />
- Component names use PascalCase.  HTML elements use lowercase.
- Displaying children using map() requires a unique key for each child.  For eg. <StoreGame key="1", gameName="myFirstGame" gameVersion="1.0" />

Day 3
- Added game cover image to games.

What I learned today:
- public vs src folder.  public/ for files that are served as is while src/ for files tied to components, are optimized by the build tool and can be imported into code.
For game cover images, better use public/ to reduce bundle size.
- inline style field does not need quotes.  "display":"flex" works same as display:"flex"
<div style={{
    display:"flex",
    flexDirection: "column"
}}>

Day 4
- Merged backend and frontend into single repo.

What I learned today:
- Whitespace inside tags don't matter, and there's white space collapsing between text elements (i.e. only 1 whitespace counts).

Day 5
- Added login link.

What I learned today:
- 3 ways to share variables:
    1. file containing global variables.  Disadv: Updating variable value will not trigger UI update, as opposed to using useState().
    2. lift state up.  Disadv: Prop drilling - Need to pass variables to child components, which is very bad for components with many nested intermediate children.
    3. use context.
- Center an element using marginLeft: "auto" and marginRight: "auto".
- Import browserRouter, routes, route in App.jsx but import Link in pages that use links.
- <div> vs <span>: <div> stacks elements vertically while <span> stacks elements horizontally.

Day 6
- Added login and sign up pages.
- Added logic to sign up button.
- Added persistent message when signed up successful.

What I learned today:
- In button's onClick props, use handleClick instead of handleClick() as handleClick() runs the function immediately, returning the value of the function.
- use e.preventDefault() to prevent form from reloading when submitting.

Day 7
- Added password validator.

What I learned today:
- 'If' conditions with lower and upper bounds do not work like in Python.  'a' <= char <= 'z' is wrong.  Instead, use ('a' <= char) && (char <= 'z')

Day 8
- Added requirements.txt to allow consistent Python package installation during local developments and CI/CD dockerization.
- Tested React client sending API request to fastAPI server locally.
- Tested sending API request to Supabase using Supabase client Python library.

What I learned today:
- Server has to define CORS policies so that it allows receiving requests from defined addresses.
- .gitignore can be defined in root directory - a separate one for each of frontend and backend is not needed.
- Store sensitive variables like API keys in .env for local developments.  Sensitive variables stored differently on Github.
- Use dotenv Python library to load .env variables.
- When creating a new virtual environment, select interpreter (Ctrl + Shift + P on vscode) to point to the Python interpreter of that venv.
Otherwise imports will remain unresolved.

Day 9
- Created sign up and login APIs on server.

What I learned today:
- Both sign up and login use POST.  Login possible using GET but it's not safe as username and password would appear in URL.

Day 10
- Set up uptimebot.com monitor to ping server and prevent server cold start.
- Defined HEAD method for "/" endpoint.

What I learned today:
- fastAPI HEAD method needs to be explicitly allowed on an endpoint to accept HEAD requests sent to it.
I initially only defined GET on the "/" endpoint and uptimebot's HEAD requests were received but rejected
by the server (error code 403).
- e.preventDefault() in form submit handlers prevents whole App() from re-rendering when the form is submitted.

Day 11
- Fixed warnings not being mutually exclusive on Sign up page.  For eg. if user enters invalid password, then signs up
successfully, both the "invalid password" and "successful sign up" would persist previously.

What I learned today:
- A JWT token helps reduce database lookups to verify user.  It contains a signature and payload, which are enough.
- localStorage and HTTPOnly cookies can both be used to persist data across page refreshes.
JS code can be injected to read from localStorage, which makes it not safe to store sensitive information like tokens.
localStorage can be used for things like dark mode toggle state.
HTTPOnly cookies are safe for storing JWT tokens.
- Passwords stored on database can be read as-is if not encrypted.  A library like bcrypt can be used to encrypt the
password so even database admin cannot make sense of it.  When logging in, same hash = same password.
